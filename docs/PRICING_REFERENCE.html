Pricing Strategy & Models

This document outlines two pricing approaches for KasamaHR: a recurring SaaS subscription model and a one-time purchase (perpetual license) model. It includes pricing, features, revenue projections, and a recommended strategy.

Pricing Models Overview

KasamaHR can be sold under two fundamentally different models. Each has distinct implications for revenue, support burden, and customer expectations.

Model A: SaaS Subscription
Per-employee, per-month recurring billing. Customers access KasamaHR as a hosted service on {company}.kasamahr.com. We handle hosting, updates, backups, and security. Three tiers differentiated by modules and limits.
Model B: One-Time Purchase
Perpetual license with a one-time fee. Customer receives the full application source code or a self-hosted deployment. They handle their own infrastructure. Optional annual support/maintenance contract for updates and priority support.

Recommendation

Strategic Recommendation Lead with SaaS (Model A) as the primary offering. Offer one-time purchase (Model B) as a premium add-on for enterprise clients who require on-premise deployment for compliance or data sovereignty reasons. This maximizes recurring revenue while capturing the on-premise market segment.

The SaaS model is recommended as the primary strategy because:

  • Predictable monthly recurring revenue (MRR) enables sustainable growth
  • Lower barrier to entry for SMEs (no large upfront cost)
  • Centralized deployment simplifies updates, support, and monitoring
  • Per-employee pricing aligns cost with value as companies grow
  • Matches the Philippine HR SaaS market trend (Sprout, GreatDay, PayrollHero)

LGU Local Government Unit Pricing

A dedicated one-time purchase model designed for Philippine Local Government Units (barangays, municipalities, cities, and provinces), structured around government procurement rules and budget cycles.

Why a Separate LGU Model LGUs have fundamentally different procurement and budgeting processes compared to private companies:
  • RA 9184 (Government Procurement Reform Act) — requires competitive bidding for purchases above 1M, with Alternative Methods of Procurement (AMP) below that threshold
  • Annual budget appropriation — funds allocated via MOOE (Maintenance and Other Operating Expenses) or Capital Outlay; no multi-year subscription commitments
  • Data sovereignty — some LGUs require data to stay on government-owned or locally hosted infrastructure
  • Different scale — employee counts range from ~20 (barangay) to 2,000+ (cities/provinces)
  • COA audit requirements — Commission on Audit needs clear documentation of software as a capital asset or service expense

LGU SaaS Subscription Option

A single, all-inclusive hosted subscription for LGUs that prefer operational expenditure (MOOE) over capital outlay. One plan, full access, no tier decisions — every LGU gets everything.

Why One Tier for LGUs Unlike the private sector where tiered pricing drives upsells, LGU procurement favors simplicity:
  • Simpler bidding — one price point means cleaner technical specifications and no confusion during BAC evaluation
  • No upgrade friction — LGU budget cycles make mid-year plan changes nearly impossible; better to give everything upfront
  • Equal access — a barangay gets the same system as a city; differentiator is employee count, not feature access
  • Faster deployment — no scoping discussions about which modules to include; deploy the full platform immediately

LGU SaaS vs. One-Time Purchase Break-Even

When does the cumulative SaaS cost exceed the one-time license? This helps LGUs choose the right model.

Sales Guidance SaaS is ideal for LGUs with fewer than ~80 employees (barangays, small municipalities) where the annual cost stays manageable and the LGU avoids the upfront capital outlay. One-time purchase is better for larger LGUs (100+ employees) where the per-employee SaaS cost exceeds the license fee within a year. Offer both and let the LGU choose based on budget type (MOOE vs. Capital Outlay) and IT capacity.

LGU SaaS Procurement Notes

Budget Classification
SaaS subscriptions are classified under MOOE → Internet/ICT Subscription Services, which is renewed annually. This is simpler to procure than Capital Outlay and doesn't require competitive bidding for amounts under 1M.
Contract Term
Annual contracts aligned to the LGU fiscal year (Jan–Dec). Auto-renews unless the LGU notifies non-renewal 30 days before term end. Multi-year contracts possible via MOA for budget certainty.
Data Sovereignty
Data is hosted on Philippine-region servers (or DICT GovCloud if required). Data export available at any time in standard formats. Full data handover upon contract termination.
Transition to One-Time
LGUs that start on SaaS can switch to a one-time license at any point. The last 12 months of SaaS payments can be credited toward the one-time license fee (up to 50% of license cost).

Pros & Cons

SaaS Subscription

  • + Predictable recurring revenue
  • + Low entry cost attracts more customers
  • + Centralized updates and security
  • + Revenue scales with customer growth
  • + Source code stays protected
  • − Ongoing hosting and infrastructure costs
  • − Churn risk (monthly cancellations)
  • − PayMongo limitations (no proration, limited methods)
  • − Must maintain 99.9%+ uptime SLA

One-Time Purchase

  • + Large upfront cash injection
  • + No churn — customer owns the license
  • + Appeals to compliance/data-sovereignty needs
  • + No ongoing hosting burden (if self-hosted)
  • − Revenue is unpredictable and non-recurring
  • − Source code delivered (IP exposure risk)
  • − Support burden for varied environments
  • − Harder to enforce updates and security patches
  • − Smaller addressable market in PH

Hybrid Combined Strategy

The recommended approach combines both models to maximize market coverage.

Hybrid Model Primary: SaaS subscription for all customers by default. Secondary: One-time purchase available only for Enterprise-tier customers upon request, positioned as a premium on-premise option with a price premium over the SaaS equivalent.

Implementation Strategy

Phase Action Timeline
Phase 1 Launch with SaaS-only. Build subscription billing, feature gating, and billing UI as described in SaaS Tier Design. Weeks 1–8
Phase 2 Begin offering one-time purchase for Enterprise prospects. Handle manually (invoice + license agreement). No code changes needed — deploy a dedicated instance per customer. Month 3+
Phase 3 If demand warrants, build self-service one-time purchase flow: license key system, download portal, auto-provisioning scripts. Evaluate based on Phase 2 volume. Month 6+ (if needed)

One-Time Purchase Delivery Model

License Agreement
Standard perpetual license with restrictions on redistribution. Customer may modify for internal use only. No resale rights. License is per-organization (not per-server).
Code Delivery
Private Git repository access (read-only) for the duration of active S&M. Alternatively, a versioned ZIP archive for customers without Git. Multi-tenant code is stripped — customer receives a single-tenant build.
Deployment Package
Docker Compose setup, .env template, database migration scripts, seed data, and a step-by-step deployment guide. Optional Ansible/Forge scripts for production deployment.
Onboarding
Enterprise Edition includes 2 hours of remote onboarding. Professional includes a 30-minute walkthrough. Starter receives documentation only.

Implementation Notes

For SaaS (Model A)

Full technical implementation is documented in SAAS_TIER_DESIGN.html. Key components:

  • PayMongo integration for recurring billing (per-tenant dynamic plans)
  • Module gating via middleware + Inertia shared props
  • FeatureGateService for centralized access checks
  • Employee count sync to update billing quantities
  • Platform admin dashboard for tenant management

For One-Time Purchase (Model B)

Minimal code changes needed for Phase 2 (manual process):

  • Create a single-tenant deployment script (strip multi-tenancy, hardcode config)
  • Build a Docker Compose production template
  • Write deployment documentation (server requirements, setup steps)
  • Draft a license agreement template
  • Set up a private repository or download delivery mechanism

Phase 3 automation (if demand warrants):

  • License key generation and validation system
  • Automated build pipeline to produce single-tenant packages
  • Customer download portal with version management
  • Auto-update notification system for S&M customers
